# memnik-os integration — schema contract + evolution playbook

> Companion to the **Sibling project: memnik-os** section of the [README](../README.md).
> The integration is **one-way**: dev-infra emits findings → memnik's bridge daemon subscribes → memnik trust spine. No code in this repo couples to memnik; the bridge implementation lives in `~/nik-memory-os/packages/server/src/integrations/dev_infra_bridge.ts`. This doc protects the WebSocket contract that the bridge depends on.

## The contract

memnik's bridge depends on **exactly this shape**. Defined in [`src/server/types.ts`](../src/server/types.ts):

### `Finding`

```ts
type Finding = {
  id: string;          // REQUIRED · stable across daemon restarts (timestamp + random suffix)
  agent: string;       // REQUIRED · agent name (e.g. "drift", "concerns-ingest")
  kind: string;        // REQUIRED · finding kind (e.g. "drift:semantic", "concern:open")
  at: number;          // REQUIRED · unix milliseconds
  severity: 'info' | 'warn' | 'error';
  summary: string;     // REQUIRED · single-line human-readable summary
  file?: string;       // OPTIONAL · when set, memnik bridge routes via observeFileChange
  line?: number;       // OPTIONAL
  suggestion?: string; // OPTIONAL · remediation hint
  payload?: Record<string, unknown>; // OPTIONAL · structured agent data
};
```

### `ServerEvent` (WebSocket frames)

```ts
type ServerEvent =
  | { type: 'finding'; finding: Finding }
  | { type: 'run'; run: AgentRun }
  | { type: 'snapshot'; findings: Finding[] };
```

Bridge consumers subscribe to `ws://127.0.0.1:5175/ws` and dedupe by `finding.id`.

## Stability guarantees

These are **load-bearing properties** the bridge relies on. Breaking any of them silently corrupts memnik's trust layer.

| Property | Guarantee | How it's enforced |
|---|---|---|
| **Finding-id stability** | Once an id is emitted, it is never re-issued or rewritten. The id is generated by `newId()` (timestamp-prefixed, random-suffixed) and persisted to `data/findings.jsonl` before the WS broadcast. Daemon restarts do not regenerate ids. | Append-only `findings.jsonl`; SQLite `findings.id` is `PRIMARY KEY`. |
| **`at` units** | Always unix milliseconds. Never seconds, never ISO-8601 strings on the wire. | `Date.now()` at emission. |
| **`severity` enum** | Exactly the three values `'info'`, `'warn'`, `'error'`. No `'debug'`, no `'critical'`, no localised variants. | Zod schema in [`src/server/agents/schemas.ts`](../src/server/agents/schemas.ts) (`SeveritySchema`). |
| **Required-field set** | `id`, `agent`, `kind`, `at`, `severity`, `summary` are present on every Finding. | Type system + schema validation per agent. |
| **127.0.0.1 binding** | Daemon binds loopback only — never publicly exposed. | Fastify boot config. |

If any of these change, **memnik's bridge breaks silently**: it'll still ingest, but produce duplicates, mis-ordered events, or corrupted timestamps that pollute the trust profile.

## Coordinated schema evolution playbook

The contract can evolve, but only on a coordinated path. Three lanes:

### Lane 1 — Adding a new optional field (backward-compatible)

Examples: a new optional `cell_urn?: string` reference, an extra `tags?: string[]` array.

- Ship in dev-infra immediately. New field appears in WS payloads.
- Memnik bridge ignores unknown optional fields by default — no breakage.
- No coordination required, but **do** open an issue / PR comment on `memnik-os` so they know the new field exists and can opt to consume it.

### Lane 2 — Adding a new `kind` value or a new agent

Examples: a new agent `auth-flow` emitting `kind: "auth:misconfigured"`.

- Ship in dev-infra. The bridge already handles arbitrary `kind` strings (it routes generically by file/agent, not by kind).
- Add the new kind to memnik's per-kind routing logic only if it deserves special handling (e.g. higher trust weight). Otherwise it lands as a generic finding.
- Coordination: optional but encouraged.

### Lane 3 — Removing or renaming a field, or changing semantics (BREAKING)

Examples: renaming `file` → `path`, dropping `payload`, changing `at` from ms to seconds, introducing a 4th severity.

**This is a coordinated migration. Both sides ship in lockstep.**

1. Open an issue in **both** repos referencing each other.
2. Agree on a deprecation window (recommended: ≥1 week).
3. Add the new field/shape **alongside** the old one in dev-infra (e.g. `path` and `file` both populated).
4. Update memnik's bridge to read the new shape and ignore the old.
5. Verify integration in both directions for the deprecation window.
6. Remove the old shape from dev-infra in a separate, clearly-titled commit.
7. Run `./scripts/verify-memnik-integration.sh` after the cutover.

**Never** change `id` stability, `at` units, or the `severity` enum — those are the bridge's hardest dependencies. If a hypothetical future need for those arises, treat it as a v2 protocol on a new endpoint (`/ws/v2`) rather than mutating `/ws`.

## Operational health checks

### Quick health check

```bash
./scripts/verify-memnik-integration.sh
```

Returns `0` if dev-infra is in a healthy state for memnik to consume. Non-zero if anything in the contract is broken.

### Manual checks

```bash
# 1. Daemon up + serving
curl -fsS http://127.0.0.1:5175/api/snapshot | jq '.findings | length'

# 2. Schema sample
curl -fsS http://127.0.0.1:5175/api/snapshot | jq '.findings[0] | keys'
# expect at minimum: ["agent","at","id","kind","severity","summary"]

# 3. Persistence (id stability check)
wc -l data/findings.jsonl
# this number must never decrease across restarts

# 4. WebSocket reachable
node -e "import('ws').then(({default:WS}) => { const w=new WS('ws://127.0.0.1:5175/ws'); w.on('open',()=>{console.log('open');w.close();}); w.on('error',e=>{console.error(e.message);process.exit(1);}); })"
```

### Recovery flows

| Symptom | Diagnosis | Recovery |
|---|---|---|
| Bridge sees no findings | Daemon down | `npm start` (or check `lsof -nP -iTCP:5175 -sTCP:LISTEN`) |
| Bridge sees duplicate findings | bridge-side dedup broken OR `id` regenerated by dev-infra | Verify last 3 ids of `findings.jsonl` are also present after restart. If yes, the bug is on the memnik side. |
| Bridge rejects findings as schema-invalid | dev-infra schema drifted | Run `./scripts/verify-memnik-integration.sh`; if it flags drift, revert the offending change. |
| Bridge connects but stalls | WS broadcast path or backpressure | Check daemon logs (`/tmp/dev-infra-restart.log` if started via the helper); restart daemon. |
| `at` values look wrong | Likely a bug emitting seconds instead of ms in a new agent | Audit any recently-added agent's emit() call; `Date.now()` is the only blessed source. |

## Why one-way

Bidirectional sync (dev-infra approvals ↔ memnik confirmations) was considered and deferred. Reasons:

- One-way is sufficient for v1. Memnik consumes; dev-infra emits. The trust layer lives on the memnik side; dev-infra has its own approval queue independently.
- Bidirectional introduces a coupling-failure mode: if either side goes down or rolls back, the other can hang on un-acked messages.
- The bridge's value (provenance + conflict surfacing) is delivered fully by the consumer side. Acks would be a polish item, not a correctness item.

If real friction shows up during dogfooding (the dual-queue causing duplicate work), revisit. Until then, one-way only.

## Where the bridge lives

| Repo | Role |
|---|---|
| `~/nik-dev-infra/` (this repo) | Emits findings on `ws://127.0.0.1:5175/ws`. Knows nothing about memnik. |
| `~/nik-memory-os/` | Bridge daemon at `packages/server/src/integrations/dev_infra_bridge.ts`. Subscribes, dedupes, routes through the trust spine. |

If you change anything WS-shaped here, run `verify-memnik-integration.sh` afterwards. If it passes, the bridge keeps working; if it fails, see the playbook above.
